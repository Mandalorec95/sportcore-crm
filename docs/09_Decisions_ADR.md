# 09. Architecture Decision Records (ADR)

## ADR-001: prisma db push вместо migrations

**Решение:** Используем `prisma db push --accept-data-loss` в Dockerfile.
**Причина:** Хакатон — быстрые итерации схемы без overhead от файлов миграций.
**Компромисс:** Данные могут быть потеряны при изменении типов/удалении полей. Для прод — migrate deploy.

## ADR-002: JWT без refresh tokens

**Решение:** Только access token (24h TTL), без refresh.
**Причина:** Упрощение архитектуры для хакатона.
**Компромисс:** Пользователи выходят раз в сутки. Для прод — добавить refresh + Redis blacklist.

## ADR-003: Многоарендность (multi-tenant) через orgId

**Решение:** Все модели содержат `orgId`. Все запросы фильтруются по `user.orgId` из JWT.
**Причина:** Изоляция данных между клубами без сложной схемы с отдельными БД.

## ADR-004: demoPassword хранится в plain text

**Решение:** При генерации пароль хешируется bcrypt и **также** сохраняется открытым в `demoPassword`.
**Причина:** Демо-режим для хакатона — нужно показать тренеру пароль ребёнка.
**Компромисс:** В продакшне это недопустимо. Нужно убрать `demoPassword` и использовать flow сброса пароля.

## ADR-005: CompetitionApproval — отдельная модель

**Решение:** Создана модель `CompetitionApproval` вместо использования `Notification.metadata`.
**Причина:** Типизированное состояние (`pending/approved/rejected/allowed_without_consent`), уникальность по `(competitionId, athleteId)`, история ответов.

## ADR-006: Оценки в Attendance, а не в ProgressRecord

**Решение:** Добавлено поле `grade Int?` в `Attendance`.
**Причина:** Оценка за конкретное занятие логически принадлежит записи посещаемости. `ProgressRecord` — для долгосрочного прогресса по навыкам.

## ADR-007: getMyChildren возвращает recentAttendance

**Решение:** `ParentsService.getMyChildren` включает последние 10 записей посещаемости с оценками.
**Причина:** Родитель должен видеть оценки без отдельного запроса.
**Компромисс:** Увеличивает payload. При большом количестве детей — пагинировать.
